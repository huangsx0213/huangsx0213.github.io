## 0. 术语

| 术语 | 定义 |
|------|------|
| **Progressive Disclosure** | 分层加载模式：元数据 → 完整内容 → 附加资源，LLM 自主决定加载深度 |
| **Interpreter Skill** | 含 `index.ts` 可执行模块的 skill；确定性逻辑由代码保证（区别于纯 prompt skill） |
| **SkillMetadata** | YAML frontmatter 中的 `name` + `description`（Anthropic Level 1） |
| **MCP Resource** | 只读、可寻址、可缓存的数据（区别于有副作用的 Tool） |
| **Deterministic 比例** | 测试用例生成过程中由代码（不靠 LLM）完成决策的比例 |
| **ReAct Loop** | 多轮 reasoning → action → observation 循环（ReAct 论文, 2022） |
| **Mode A / Mode B** | 确定性 Pipeline / 自主 Orchestrator（见 2026-05-31 spec） |

## 1. 背景与动机

### 1.1 当前 Mode A 实测加载量

| Agent | `requiredSkills` | SKILL.md 全文 | references 全文 | **合计 chars** | **折算 tokens** |
|-------|-----------------|--------------|----------------|---------------|----------------|
| `TestAnalystRole` | 5 个 | 24,551 | 59,036（`requirement-index/references/index.json`） | **~83,587** | **~21,000** |
| `TestDesignerRole` | 2 个 | 18,034 | 0 | **~18,034** | **~4,500** |
| `QualityManagerRole` | 2 个 | 18,316 | 0 | **~18,316** | **~4,600** |
| **3 agent × N batch** | — | — | — | — | **~30,100 × N tokens** |

> 关键发现：`requirement-index/references/index.json` 单文件 59KB，每次 Analyst 调用都全量塞进 system prompt。

### 1.2 三个症状

1. **Token 浪费**：每次调用都加载所有 `requiredSkills` 全文 + 所有 references
2. **违反 Progressive Disclosure 原则**：3 个 sub-agent role 的 `useProgressiveDisclosure` 默认 `false`，全量加载
3. **按需加载工具不可用**：`allowedTools: ['execute_skill_module']` 已声明，但 `useReActLoop: false` 关闭了工具暴露

### 1.3 文档目标

1. 调研业界 5 套主流范式（Anthropic Skills、MCP、LangGraph、Qodo/MTSC、LangChain Skills）
2. 与当前项目对标，定位差距
3. 提出 3 阶段演进路径（保守 → 渐进 → 分层架构）

## 2. 业界 5 套主流范式

### 2.1 范式 1：Anthropic Skills 三层渐进披露

> 来源：[Anthropic: Equipping agents for the real world with Agent Skills](https://claude.com/blog/equipping-agents-for-the-real-world-with-agent-skills)（2025-10-16 官方发布）
> 配套文档：[The Complete Guide to Building Skills for Claude (PDF)](https://resources.anthropic.com/hubfs/The-Complete-Guide-to-Building-Skill-for-Claude.pdf?hsLang=en)

**三层架构**：

```
┌────────────────────────────────────────────────────────────┐
│ Level 1: YAML frontmatter                                  │
│   - 始终注入 system prompt                                  │
│   - 只含 name + description                                 │
│   - ~100 tokens/skill                                       │
├────────────────────────────────────────────────────────────┤
│ Level 2: SKILL.md body                                     │
│   - LLM 通过 load_skill tool 决定加载                       │
│   - 完整指令、决策表、样例                                   │
│   - <5k tokens                                              │
├────────────────────────────────────────────────────────────┤
│ Level 3: references/ 链接文件                              │
│   - LLM 进一步按需导航读取                                  │
│   - 详细 schema、form-filling 规则等                         │
└────────────────────────────────────────────────────────────┘
```

**核心设计原则**：

| 原则 | 含义 | 业界示例 |
|------|------|---------|
| **Skills = 文件夹** | 包含 `SKILL.md` + `scripts/`（**可执行代码**）+ `references/` + `assets/`（模板） | `anthropics/skills` GitHub 仓库 |
| **Description 写法** | 写"**什么时候用**"，不是"能做什么" | Anthropic 内部 100+ skill |
| **Gotchas 段落** | 记录 Claude 实际踩过的坑（最值钱） | `frontend-design/SKILL.md` |
| **代码嵌入** | 给 Claude 可调用 Python/Bash helper，**释放 LLM 决策带宽** | `pdf` skill 填表 |
| **Composability** | 多 skill 同时待命，按需激活 | `signup-flow-driver` + `tmux-cli-driver` |
| **Open Standard** | 2025-12-18 发布为 `agentskills.io` 开放标准 | 跨平台（Claude.ai / Claude Code / API） |

**Token 经济**（官方数据）：

| 阶段 | Token 占用 | 时机 |
|------|----------|------|
| 启动 | 7 × 100 = ~700 tokens | 立即 |
| Analyst 激活 | +3k tokens | 调用时 |
| Requirement 详细查询 | +5-50k tokens | LLM 决策触发 |
| **总量（典型）** | **~10-60k tokens** | 取决于任务 |

> 对比当前项目：~21k tokens/Analyst 调用，且不含 "按需" 优化空间。

### 2.2 范式 2：MCP（Model Context Protocol）— Read vs Do 分离

> 来源：[MCP Tools 规范 (2025-11-25)](https://modelcontextprotocol.io/specification/2025-11-25/server/tools) + [Client Best Practices: Progressive Tool Discovery](https://modelcontextprotocol.io/docs/develop/clients/client-best-practices) + [Microsoft: MCP Resources as Tool Inputs](http://microsoft.github.io/mcscatblog/posts/mcp-resources-as-tool-inputs/)

**资源/工具二维分类**：

```
┌────────────────────────┬────────────────────────┐
│  Tools (执行动作)       │  Resources (读数据)     │
├────────────────────────┼────────────────────────┤
│ 有副作用 (写 DB/文件)   │ 只读、无副作用          │
│ LLM 决策调用            │ Client/用户主动拉取     │
│ Output 不可缓存         │ 可订阅、可缓存          │
│ Token 影响: 不确定       │ Token 影响: 已知        │
│ 例: queryDB/apiCall    │ 例: db://schema.json   │
└────────────────────────┴────────────────────────┘
```

**Progressive Tool Discovery**（官方推荐模式）：

> "The host fetches tool definitions via `tools/list` as normal, but defers injecting them into the model's context. The host provides a lightweight `search_tools` meta-tool. The host loads full definitions into context only as needed."

**触发阈值建议**（官方）：
- 工具定义占总 context < 1-5% → 全部加载 OK
- 超过阈值 → 切到 progressive discovery
- 实现 search strategy：BM25 / 向量相似 / subagent-based（Haiku/Flash 选 tool）

**核心反模式**（[apigene.ai 56 个开发者讨论分析](https://apigene.ai/blog/mcp-resources)）：

| 反模式 | 后果 | 正确做法 |
|--------|------|---------|
| Tool 输出 30k chars 全量塞 LLM | context 爆 | Server 存数据，agent 传 URI |
| 98% tool description 没写"何时用" | LLM 不会选 | 改写 description（何时用） |
| Tool-only 架构 | token 燃烧 | 引入 resources 分层 |
| 静态加载所有 tool | 启动慢 | Dynamic loading 省 12k tokens/session |

### 2.3 范式 3：LangGraph Subgraph as Node vs as Tool

> 来源：[LangChain Forum 官方回复](https://forum.langchain.com/t/how-to-use-a-subgraph-as-a-tool/2175/1) + [LangGraph Tool Calling Docs](https://langchain-ai.github.io/langgraph/how-tos/tool-calling/)

**两种集成模式**：

```
Option A: 紧集成 — Subgraph 作为节点
  ✓ 共享 checkpoints、interrupts、time-travel
  ✓ Parent state 直接 update
  ✓ 适合：固定流水线 (A → B → C)
  例：当前 Mode A 的 3 agent + 3 checkpoint 串行

Option B: 灵活 — Subgraph 作为 Tool
  ✓ ReAct agent 自主决定何时调用
  ✓ LLM-driven 路由
  ✓ 适合：探索性、动态决策
  例：Mode B Orchestrator 的 spawn_subagent
```

**当前项目使用情况**：

| 模式 | 用在哪 | 状态 |
|------|-------|------|
| Option A | Mode A 流水线 | ✅ 已实现 |
| Option B | Mode B Orchestrator | ⚠️ 部分（`spawn_subagent` 已写，但未实测） |
| Option B | Mode A sub-agent | ❌ 未实现（`useReActLoop: false` 关闭了） |

### 2.5 范式 5：LangChain Skills SQL Assistant 范式

> 来源：[LangChain 官方教程: Build a SQL assistant with on-demand skills](https://docs.langchain.com/oss/javascript/langchain/multi-agent/skills-sql-assistant)

**架构示意**：

```typescript
// 1. Skill metadata 始终在 system prompt
const systemPrompt = skills.map(s => 
  `- ${s.name}: ${s.description}`
).join('\n');

// 2. load_skill tool — 加载完整 SKILL.md
const loadSkill = tool(
  async ({ skillName }, runtime) => {
    const skill = await loadSkillContent(skillName);
    return Command(update={{
      messages: [new ToolMessage(skill.content, runtime.tool_call_id)],
      loadedSkills: [...state.loadedSkills, skillName],
    }});
  },
);

// 3. Skill middleware — 注入已加载 skill
const skillMiddleware = createMiddleware({
  wrapModelCall: (request, handler) => {
    const skills = state.loadedSkills.map(loadFullSkill);
    request.systemPrompt = `${request.systemPrompt}\n\n${skills.join('\n')}`;
    return handler(request);
  },
});

// 4. Agent 通过 tool call 决定何时加载
const agent = createAgent({
  model: 'anthropic:claude-3-7-sonnet',
  tools: [loadSkill, ...otherTools],
  middleware: [skillMiddleware],
});
```

**Progressive Disclosure 4 大动机**（官方原文）：
1. **Reduce context usage** — 只加载 2-3 个相关 skill
2. **Team autonomy** — 不同团队维护不同 skill
3. **Scales efficiently** — 数十上百个 skill 不爆 context
4. **Simplify history** — 单 agent 单 thread

## 4. 核心洞察

### 4.1 决策类型应该放哪里

| 决策类型 | 应放哪 | 理由 |
|---------|-------|------|
| **ISTQB 技术选择**（EP/BVA/DT/ST/UC） | **Skill 的 `index.ts`（代码）** | 决定论、可测试、零 LLM 成本 |
| **覆盖维度枚举**（输入域） | **Skill 的 `index.ts`** | 决定论（BVA 自动生成边界值） |
| **风险评级** | **`index.ts`** + LLM 综合 | 部分可代码化（critical → high） |
| **步骤拆解**（precondition/action/expected） | **LLM** | 需要语义理解 |
| **步骤排序/原子性** | **`index.ts`** | 决定论（sequence 自动递增） |
| **自然语言表达** | **LLM** | 需要语言能力 |
| **HITL 反馈应用** | **`index.ts`** 中间件 | 决定论（保留 id、合并新增） |

### 4.2 Deterministic 化比例目标

| 目标 | 比例 | 适用场景 |
|------|------|---------|
| 保守 | 30% | 紧急修 token 问题（Phase 1） |
| 平衡 | 50% | Qodo 水平（Phase 2） |
| 理想 | 80% | testRigor / Diffblue 思路（Phase 3） |

### 4.3 Token 节省估算

| 阶段 | 当前 | Phase 1 后 | Phase 2 后 |
|------|------|----------|----------|
| Analyst 启动 | 21k | ~700（metadata） | ~700 |
| Analyst 按需加载 | — | — | ~3-5k（按需 EP/BVA） |
| 总量（典型） | 21k | ~5k | ~3-8k |
| **节省** | — | **-76%** | **-62% ~ -86%** |

### 4.4 重开 ReAct 的风险与对冲

| 风险 | 来源 | 对冲方案 |
|------|------|---------|
| 工具调用风暴 | LLM 过度调 execute_skill_module | `maxIterations` 限制（如 3 轮） |
| 数据流再次断裂 | Tool output 嵌套问题 | 已修复 [`tool.ts`](../shared/ai/tool.ts)（data 不再嵌套） |
| 成本不可控 | 多轮 LLM 调用 | TokenTracker 已在 `ReactLoopState` 累积 |
| 行为不可预测 | LLM 决策 | 关键决策下沉到 `index.ts`（deterministic） |

## 5. 建议方案：3 阶段演进

### Phase 1：保守型（立即可做，零风险）

**目标**：仅启用 metadata-only 加载，**修 token 问题，不改数据流**。

**改动**：

```typescript

  // ... 现有字段

};

// shared/ai/roles/test-designer.ts
export const TestDesignerRole: AgentRole = {
  // ... 现有字段
  useProgressiveDisclosure: true,  // ← 新增
};

// shared/ai/roles/quality-manager.ts
export const QualityManagerRole: AgentRole = {
  // ... 现有字段
  useProgressiveDisclosure: true,  // ← 新增
};
```

**验证**：

```typescript
// 单元测试
it('Mode A sub-agents use metadata-only loading', () => {
  expect(TestAnalystRole.useProgressiveDisclosure).toBe(true);
  expect(TestDesignerRole.useProgressiveDisclosure).toBe(true);
  expect(QualityManagerRole.useProgressiveDisclosure).toBe(true);
});

it('Token usage reduced >70% with useProgressiveDisclosure=true', () => {
  const oldCtx = createAgentContext(provider, { ...TestAnalystRole, useProgressiveDisclosure: false });
  const newCtx = createAgentContext(provider, { ...TestAnalystRole, useProgressiveDisclosure: true });
  expect(newCtx.skillContext.systemPrompt.length).toBeLessThan(oldCtx.skillContext.systemPrompt.length * 0.3);

```

**预期收益**：
- Analyst 21k tokens → ~700 tokens（**-97%**）
- Designer 4.5k → ~300（**-93%**）
- Quality 4.6k → ~300（**-93%**）
- 风险：极低（只影响 system prompt 注入方式）

**已知 trade-off**：
- LLM 看不到 SKILL.md 全文 → 决策质量可能下降
- 解决方案：Phase 2 补上"按需 load_skill 工具"

**工期**：< 1 小时（含测试）

```typescript
import { z } from 'zod';

// ISTQB 5 种技术 → 自动生成 technique skeleton
export const techniques = {
  'equivalence-partitioning': (cond) => generateEpCases(cond),
  'boundary-value-analysis': (cond) => generateBvaCases(cond),
  'decision-table': (cond) => generateDtCases(cond),
  'state-transition': (cond) => generateStCases(cond),
  'use-case': (cond) => generateUcCases(cond),
} as const;

// BVA: 输入域边界值自动展开（deterministic）
export function generateBvaCases(condition: Condition): TestCase[] {
  const { dimension, variants } = condition.coverageDimensions[0];
  const boundary = computeBoundary(dimension);  // min, min-1, min+1, max-1, max, max+1
  return boundary.map(value => ({
    id: `${condition.id}-bva-${value}`,
    techniqueApplied: 'boundary-value-analysis',
    testData: [{ name: dimension, value, type: inferType(value) }],
    steps: generateAtomicSteps(condition, value),
  }));
}

// 风险评级（deterministic + LLM 综合）
export function inferRiskLevel(requirement: Requirement): RiskLevel {
  if (requirement.priority === 'CRITICAL') return 'high';
  if (requirement.priority === 'HIGH' && hasPaymentOrAuth(requirement)) return 'high';
  return 'medium';
}
```

- 与 Anthropic Skills 规范对齐（`scripts/` 目录）

**风险**：
- ReAct 工具调用风暴 → `maxIterations: 3` 限制
- 数据流回退 → 保留 `useReActLoop: false` 旁路

**工期**：2-3 天（含 5 个 skill 的 index.ts 编写 + 单测）

### Phase 3：分层架构（MCP-style，可选长期目标）

**目标**：把 `references/` 改为可寻址 resource，按需 fetch。

**改动**：

```
skills/requirement-index/
  SKILL.md
  index.ts
  resources/             ← 改名为 resources/
    requirement-{epicId}.json  ← 按 epic 拆分（不再 1 个 59KB 文件）
    query-grammar.md
  resource-manifest.ts    ← 新增：resource 注册表
```

**`resource-manifest.ts` 示例**：

```typescript
export const resources = {
  'requirement-epic-123': {
    uri: 'resource://requirement-index/requirement-123.json',
    type: 'json',
    size: 4096,
    description: 'Requirements for epic 123 (Login & Auth)',
  },
  'requirement-epic-124': { /* ... */ },
} as const;
```

**Tool 升级为 resource-aware**：

```typescript
const fetchRequirementResource = tool({
  name: 'fetch_requirement_resource',
  description: 'Fetch a specific requirement resource by URI. Use this when the requirement-epic summary is not enough.',
  schema: z.object({ uri: z.string() }),
  execute: async ({ uri }) => {
    // 返回 resource 内容（不返回大段 JSON）
  },
});
```

**预期收益**：
- 完全对齐 MCP 规范
- 资源可订阅、可缓存
- 适配 100+ skill 不爆 context

**工期**：1 周（架构改造 + 现有 skill 迁移）
